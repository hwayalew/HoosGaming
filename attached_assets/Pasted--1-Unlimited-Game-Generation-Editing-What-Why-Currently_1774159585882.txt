#1 - Unlimited Game Generation & Editing
What & Why
Currently the app sends one IBM watsonx prompt and expects the full game in a single response. This hard-limits output to ~6k–9k characters — nowhere near enough for a complete, full-featured game. Users should be able to describe any game, from a simple prototype to a full multi-level title, and get working code that runs without being cut short. The AI should decide how long the code needs to be with no imposed ceiling.

Done looks like
The AI generates as much code as the game needs. If the response is cut off mid-generation, the system automatically fires a continuation on the same IBM thread ("Continue exactly where you left off, output only the continuation") and keeps going until the game code is complete and valid — no hard pass limit, just a generous safety ceiling (e.g., 20 passes) to prevent runaway loops.
Completion is determined by code validity: closed </html> tag, presence of the Phaser.Game / Three.js render loop / Pyodide bootstrap call, and no open function or class blocks.
All continuation chunks are assembled into a single clean HTML file before being sent to the client, with duplicate headers removed and script content merged correctly.
The assembled game renders and runs correctly in the /play iframe regardless of how many passes were needed.
After a game is generated, users can send follow-up messages (e.g., "add a second boss", "add a shop menu", "add a new level") and the system will apply those changes using the same thread context. The modification also uses the auto-continuation loop so edits to large games are never truncated either.
The Create page shows live progress: current pass number, a status label ("Generating…", "Continuing…", "Assembling…"), and a running character count so the user can see the game growing in real time.
The /play page renders the final game at full browser size and supports fullscreen.
Out of scope
Streaming token-by-token output from IBM (polling API only, not SSE).
Persisting games to a database between sessions.
Splitting the game across multiple files or modules (output is always a single self-contained HTML file).
Tasks
Completion detection utility — Write a function that inspects an accumulated code string and returns whether the game is complete: checks for a closed </html> tag, at least one game bootstrap call (new Phaser.Game, renderer.render, pyodide.runPythonAsync), and no unmatched open braces indicating a cut-off function or class. This is the loop's exit condition.

Unlimited auto-continuation loop — Replace the single IBM call in the chat API route with a loop. After each response, run the completion check. If incomplete, send a bare continuation prompt on the same thread and append the result. Repeat until complete or a generous safety limit (20 passes) is reached. No user action required between passes.

Code assembly — After the loop, merge all chunks into one valid HTML string: keep the <!DOCTYPE html> header only from the first chunk, strip repeated opening tags from continuations, join all <script> body content in order, and ensure the file closes with </html>. Run fixCensoredUrls over the assembled output.

Modification support for large games — When the user sends a follow-up edit request (e.g., "add a new enemy type"), pass the full assembled game code as context in the IBM thread and apply the same auto-continuation loop to the modification response. This ensures edits to large games are also never cut short.

Live progress UI — On the Create page, show the current pass number, a status label, and a live character count that updates as each chunk arrives ("Pass 3 — 18,400 chars generated…"). This gives the user confidence that a long generation is still running, not stalled.
